❌ Traditional Thread Approach

Each network call creates a new OS thread. Threads are expensive — each holds ~1MB of stack memory and the OS must context-switch between them.

Thread-1 (UI)
BLOCKED — waiting for network...
Thread-2 (Network)
BLOCKED — waiting for DB...
Thread-3 (DB)
Doing actual work
// Traditional Thread — callbacks, nested, hard to read
fun loadUserProfile() {
    Thread {
        val user = api.getUser() // BLOCKS this thread
        Thread {
            val posts = db.getPosts(user.id) // BLOCKS another thread
            runOnUiThread { updateUI(user, posts) } // Callback hell
        }.start()
    }.start()
}

✅ Coroutine Approach

One thread handles many coroutines. When a coroutine suspends (waiting for I/O), the thread is freed to execute other coroutines. No blocking, no waste.

Main Thread
✓ Free — serves UI & other coroutines
IO Thread
✓ Free when awaiting response
// Coroutine — sequential, readable, non-blocking
fun loadUserProfile() {
    viewModelScope.launch {
        val user = withContext(Dispatchers.IO) { api.getUser() } // suspends, not blocks
        val posts = withContext(Dispatchers.IO) { db.getPosts(user.id) }
        updateUI(user, posts) // back on Main automatically
    }
}

🚀 launch — Fire and Forget

Returns a Job. No result. Exceptions propagate immediately to the scope.

val job: Job = viewModelScope.launch {
    analyticsService.logEvent("screen_open")
    // returns nothing — fire and forget
}

// You can cancel it
job.cancel()
1
Coroutine starts immediately in viewModelScope
2
Executes logEvent() — suspends if needed
3
Completes. Returns Job that you can .cancel()

⚡ async — Concurrent Result

Returns a Deferred<T>. Call .await() to get the result. Best for parallel operations.

// Fetch two APIs concurrently — much faster than sequential
viewModelScope.launch {
    val userDeferred = async(Dispatchers.IO) { api.getUser() }
    val postsDeferred = async(Dispatchers.IO) { api.getPosts() }

    // Both run in parallel — total time = max(user, posts), not sum
    val user = userDeferred.await()
    val posts = postsDeferred.await()
    updateUI(user, posts)
}

⚠️ Why GlobalScope is Dangerous

GlobalScope lives for the entire app lifetime. It is not tied to any component's lifecycle, causing memory leaks and zombie coroutines.

⛔ MEMORY LEAK
// User opens UserActivity → coroutine starts
class UserActivity : AppCompatActivity() {
    override fun onCreate(...) {
        GlobalScope.launch {  // ⚠️ NOT tied to Activity lifecycle
            val data = api.getHeavyData() // Takes 10 seconds
            updateUI(data) // ⚠️ Activity may be dead here!
        }
    }
}
// User presses back → Activity destroyed
// GlobalScope coroutine KEEPS RUNNING for 10 more seconds 💀
// Then crashes trying to update a dead UI reference
✅ CORRECT
class UserActivity : AppCompatActivity() {
    override fun onCreate(...) {
        lifecycleScope.launch {  // ✅ Tied to Activity lifecycle
            val data = api.getHeavyData()
            updateUI(data) // ✅ Safe — auto-cancelled if Activity dies
        }
    }
}
// User presses back → Activity destroyed → coroutine CANCELLED ✅

⛔ GlobalScope

  • Lives for the full app lifetime
  • Not cancelled on back press
  • Causes memory leaks
  • Can crash destroyed UI
  • Impossible to unit test

✅ Structured Scopes

  • viewModelScope — tied to ViewModel
  • lifecycleScope — tied to Activity/Fragment
  • Auto-cancelled when owner dies
  • Zero memory leaks
  • Easily testable with TestScope

🔄 Cooperative Cancellation

Cancellation in coroutines is cooperative. A coroutine must periodically check if it's been cancelled. Suspension points (like delay) check automatically.

val job = viewModelScope.launch {
    for (i in 1..100) {
        // ✅ isActive check — cooperative with cancellation
        if (!isActive) return@launch

        processChunk(i) // heavy CPU work
    }
}

// Later — user navigates away
job.cancel() // signals cancellation — coroutine will stop at next isActive check
1
job.cancel() is called — sets cancellation flag on the Job
2
Next suspension point (delay, yield, withContext) throws CancellationException
3
For CPU loops, check isActive manually — it's cooperative, not forced
4
try/finally still runs — you can clean up resources safely
// Always wrap in try/finally for cleanup
viewModelScope.launch {
    try {
        delay(5000) // auto-checks cancellation here
        updateUI()
    } finally {
        // ✅ This always runs — close streams, hide loaders etc.
        hideLoadingSpinner()
    }
}

🚦 Dispatchers — Where Coroutines Run

M
Dispatchers.Main — UI thread only
Use for: updating views, observing StateFlow, Compose recompositions
IO
Dispatchers.IO — thread pool (up to 64 threads)
Use for: Retrofit calls, Room queries, file read/write, SharedPrefs
D
Dispatchers.Default — CPU thread pool (= CPU core count)
Use for: JSON parsing, image processing, sorting large lists
U
Dispatchers.Unconfined — runs in caller's thread (testing only)
Avoid in production — unpredictable thread behavior 
// Pattern: start on Main, switch to IO, return to Main
viewModelScope.launch {                  // starts on Main
    showLoading()                              // Main — updates UI

    val result = withContext(Dispatchers.IO) {
        repository.fetchData()              // IO — network call
    }                                          // automatically back on Main

    hideLoading()                              // Main — updates UI
    render(result)
}

✓ withContext automatically restores the original dispatcher after the block finishes.
